# Fleet v0 completion audit

**Audit date:** 2026-07-15  
**Specification:** [spec-v0.md](spec-v0.md)  
**Product contract:** [product-vision.md](product-vision.md)

This document maps the normative v0 requirements to current implementation and runtime evidence. A requirement is considered satisfied only where the repository or recorded execution proves it. Optional mixed macOS/Linux mutation tests are listed separately and are not part of the v0 completion bar, per the product owner's decision not to require destructive reconfiguration of their only Mac.

## Product boundary and invariants

| Requirement | Implementation/evidence | Result |
|---|---|---|
| Fleet supplies device fabric and does not delegate, schedule, or monitor work. | Only `init`, `join`, `status`, and `leave` are public. The daemon exposes identity, join, and leave registration routes only. The Fleet skill tells agents that the user chooses the target. | Satisfied |
| A machine is uninitialized, captain, or member of exactly one fleet. | `LocalConfig` has one `Role`; state validation rejects invalid role/captain shapes. Existing incompatible state routes users to `fleet leave`. | Satisfied |
| A fleet has no separate name. | No fleet-name field, prompt, state, or command exists. The captain anchors topology. | Satisfied |
| IP addresses are not durable identity. | Durable records contain UUIDs, Ed25519 keys/fingerprints, names, and `.local` hosts. Inventory and skill contain no IP fields. Network endpoints exist only during requests/discovery. | Satisfied |
| Names use the conservative 1–63 character lowercase grammar. | `validate_machine_name` is used for local input, persisted state, and network registrations; boundary cases have unit coverage. | Satisfied |
| Trust is captain-to-member only for the joining user. | A dedicated captain Ed25519 key is installed only into the current account's `authorized_keys`. Fleet creates no reverse/member-to-member/root trust. `id -un` determines the real current user. | Satisfied |

## Files and state

| Requirement | Implementation/evidence | Result |

| Private material and directories have restrictive permissions. | Identity directory is `0700`; private key is generated by OpenSSH; config, inventory, authorized keys, known hosts, and generated SSH config are `0600`. | Satisfied |
| Writes that affect trust or membership are atomic. | `state::atomic_write` writes, syncs, sets permissions, and persists a same-directory temporary file. It is used for config, inventory, SSH trust/config, shell integration, and skill references. | Satisfied |

## Tests and runtime proof

The completion gate is:

```sh
cargo fmt --check
cargo clippy --all-targets --all-features -- -D warnings
cargo test
cargo test --test macos_bonjour -- --ignored --nocapture
sh -n fleet.sh
```

Current automated coverage includes state/version/role validation, name grammar, key canonicalization and fingerprinting, interrupted identity recovery, discovery identity consistency, registration input/pin validation, shell/rc behavior and permissions, command plans for both platforms, LaunchAgent/systemd rendering, tmux guards/contrast, skill generation/removal, OpenSSH include ordering/removal, known-host regeneration, CLI errors/idempotency, and real native Bonjour publish/browse.

The final Linux acceptance rerun used the two existing Ubuntu 24.04 ARM64 Multipass machines, each constrained to 2 CPUs, 2 GiB RAM, and 10 GiB storage. It proved real mDNS join, real hostname changes, strict passwordless SSH, effective plain-host SSH configuration, tool detection, member leave, captain leave, cleanup, and preservation. Both VMs were stopped afterward. Full command-level evidence is recorded in [verification.md](verification.md).

## Explicitly deferred and absent

The v0 public CLI and implementation do not contain Tailscale, off-LAN access, sync/transfer, skills on members, task delegation/scheduling, T3 Code setup, additional shells, Windows/WSL, multi-fleet membership, captain election/recovery, or continuous monitoring. The hidden daemon and diagnostic endpoint option support only the v0 lifecycle and testing.

## Optional post-v0 confidence work

If a Mac that may safely be renamed becomes available, the two mixed macOS/Linux directions can be exercised. They are not needed to establish v0 completion and must not be run against the product owner's only Mac merely to increase test coverage.
